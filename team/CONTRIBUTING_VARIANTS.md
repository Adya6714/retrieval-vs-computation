# Contributing Variants ‚Äî Style Guide

Read this entire document before writing any variant. Every variant goes through cross-review before entering the CSV. If you are unsure about anything, flag it ‚Äî do not guess.

---

## What a variant is

A variant is a rewritten version of a canonical problem. The correct answer must be identical to the canonical problem. What changes is the surface form ‚Äî how the problem is expressed, not what the problem is asking.

There are six variant types. Each canonical problem gets a subset of these depending on which family it belongs to (see applicability table below).

---

## The six variant types

### W1 ‚Äî Lexical Paraphrase
**What changes:** Vocabulary and sentence structure. The problem is completely rewritten in different words.
**What stays the same:** All numbers, all constraints, the correct answer, the difficulty.
**What must NOT change:** Any quantity, any relationship between elements, the question being asked.

This is not synonym substitution. "The edge from A to B has weight 3" rewritten as "The connection from A to B weighs 3" is not W1 ‚Äî that is a surface swap. W1 means genuinely rewriting the sentence structure. "A path connects node A to node B at a cost of 3 units" is W1.

**Common mistake:** Changing numbers accidentally while rewriting. Check every number against the original before submitting.

---

### W2 ‚Äî Structural Reformat
**What changes:** The presentation format. Prose becomes a table, a JSON object, or a structured list.
**What stays the same:** All information, all numbers, the question.
**What must NOT change:** Any quantity, any relationship, any constraint.

The reformatted version must contain all the information in the original. Do not summarize. Do not drop any edge, constraint, or quantity.

**Example ‚Äî Shortest Path:**

Original (prose):
"There is a network with nodes A, B, C, and G. The edge from A to B costs 3. The edge from A to C costs 1. The edge from B to G costs 4. The edge from C to G costs 6. What is the shortest path from A to G?"

W2 (table):
```
| From | To | Cost |
|------|----|------|
| A    | B  |    3 |
| A    | C  |    1 |
| B    | G  |    4 |
| C    | G  |    6 |

Find the path from A to G with minimum total cost.
```

W2 (JSON):
```json
{
  "nodes": ["A", "B", "C", "G"],
  "edges": [
    {"from": "A", "to": "B", "cost": 3},
    {"from": "A", "to": "C", "cost": 1},
    {"from": "B", "to": "G", "cost": 4},
    {"from": "C", "to": "G", "cost": 6}
  ],
  "query": "Find the path from A to G with minimum total cost."
}
```

**Example ‚Äî Blocksworld:**

Original:
"Block A is on block B. Block B is on the table. Block C is on the table. Your goal is to have C on B and B on A. What moves do you make?"

W2 (structured):
```
Initial state:
- A is on B
- B is on table
- C is on table

Goal state:
- C is on B
- B is on A

What sequence of moves achieves the goal?
```

### W3 ‚Äî Entity Rename
**What changes:** All named labels (node names, block names, variable names) are replaced with different labels.
**What stays the same:** The structure, all numbers, all relationships, the correct answer.
**What must NOT change:** The topology, the costs, the constraints.

The mapping must be consistent throughout. If node A maps to "warehouse_7", every occurrence of A becomes "warehouse_7". Never partial ‚Äî if you rename some labels but not others the problem becomes inconsistent.

**Example ‚Äî Shortest Path:**

Original: nodes A, B, C, G with edge costs as above.
W3: replace A‚Üíserver_1, B‚Üíserver_2, C‚Üíserver_3, G‚Üíserver_4. All edge costs identical. Query asks for shortest path from server_1 to server_4.

**Example ‚Äî Blocksworld:**

Original: blocks A, B, C.
W3: replace A‚Üíred, B‚Üíblue, C‚Üígreen. Initial and goal states use new names throughout.

**Common mistake:** Forgetting to rename a label in the question at the end.

---

### W4 ‚Äî Formal Notation
**What changes:** Natural language is replaced with mathematical or formal notation.
**What stays the same:** The problem semantics, all quantities, the correct answer.
**What must NOT change:** Any constraint or relationship.

This variant only applies to families where formal notation has a clean counterpart (see applicability table). Do not force formal notation onto families where it distorts the problem.

**Example ‚Äî Shortest Path:**

Original: prose description of graph.
W4: Express as a weighted graph G = (V, E, w) where V = {A, B, C, G}, E = {(A,B), (A,C), (B,G), (C,G)}, w(A,B)=3, w(A,C)=1, w(B,G)=4, w(C,G)=6. Find the path p* = argmin_{p: A‚ÜíG} Œ£ w(e) for e in p.

**Example ‚Äî Blocksworld:**

Original: natural language state description.
W4: Express in PDDL notation with explicit (on X Y) and (ontable X) predicates for initial and goal states.

---

### W6 ‚Äî Procedural Regeneration
**What changes:** Everything ‚Äî new numbers, new parameters, new instance. Generated by a script with a documented random seed.
**What stays the same:** The problem structure and type (same algorithm, same family).
**What must NOT change:** The structural properties that define the difficulty tier.

This variant is not written by hand. Adya owns the generation scripts. Your job is to verify that the generated instance is valid: correct answer is correct, difficulty tier is appropriate, answer is unambiguous.

**How to verify a W6 instance:**
- Solve it yourself independently. Does your answer match the script's stated correct answer?
- Is the difficulty tier consistent with the criteria in `INSTANCE_SELECTION_CRITERIA.md`?
- For Blocksworld: simulate the plan step by step. Does it reach the goal state?
- For algorithmic: run the verifier function. Does it confirm the answer?

If any check fails, flag it ‚Äî do not submit.

---

### W5 ‚Äî Reversal
**What changes:** The direction of the problem. The question is flipped.
**What stays the same:** The underlying structure.
**What changes in the answer:** The correct answer is different from the canonical problem. This is intentional and expected.

W5 is the only variant where the correct answer changes. Report W5 accuracy separately ‚Äî it is never pooled with W1‚ÄìW5 in the main CSS computation.

**Applies to:** Blocksworld only (within Planning Suite), Shortest Path and Coin Change only (within Algorithmic Suite). Does not apply to Arithmetic or other sub-types.

**Example ‚Äî Shortest Path reversal:**
Original: find shortest path from A to G.
W5: find shortest path from G to A. (For undirected graphs the answer is the same path reversed. For directed graphs the answer may differ or the path may not exist ‚Äî flag ambiguous cases.)

**Example ‚Äî Blocksworld reversal:**
Original: given initial state S, reach goal state G.
W5: given initial state G (the original goal), reach state S (the original initial state). The correct plan is the reverse sequence of moves ‚Äî but only if every reversed move is legal. Verify this before submitting.

**Important:** Always verify that the reversed problem is well-defined and has a unique answer before submitting. If the reversal is ambiguous, flag it.

## Variant applicability by family

| Variant | Planning Suite | Arithmetic Reasoning | Algorithmic Suite |
|---------|----------------|----------------------|-------------------|
| W1 | All sub-types | All | All |
| W2 | All sub-types | All | All |
| W3 | All sub-types | All | All |
| W4 | BW + Logistics only | Not applicable | All |
| W6 | All sub-types | All | All |
| W5 | Blocksworld only | Not applicable | SP + Coin Change only |

Mystery Blocksworld: W4 does not apply (already a form of obfuscation ‚Äî applying formal notation on top is redundant). W5 does not apply cleanly.

## The variants CSV

All variants go in `data/problems/probe1_variants.csv`.

| Column | What to put |
|--------|-------------|
| `problem_id` | Same as the canonical problem this is based on |
| `variant_id` | `BW_001_W1`, `SP_003_W4`, etc. |
| `variant_type` | `W1`, `W2`, `W3`, `W4`, `W5`, `W6` |
| `variant_text` | Full problem statement of the variant |
| `correct_answer` | Same as canonical for W1‚ÄìW4, W6. Different (and explicitly computed) for W5. |
| `written_by` | Your name |
| `reviewed_by` | Leave blank until review is complete |
| `review_status` | `pending`, `approved`, `needs_fix` |

## Pre-submission checklist

Before submitting any variant for review:

- [ ] Correct answer is identical to canonical (W1‚ÄìW4, W6), or explicitly computed and verified (W5)
- [ ] Every number in the original appears unchanged in the variant (W1, W2, W3, W4)
- [ ] Entity mapping is consistent throughout (W3)
- [ ] No information was dropped in reformatting (W2)
- [ ] No LLM was used to generate this variant
- [ ] Another team member can independently compute the same answer from this variant
- [ ] `variant_id` follows the format `PROBLEMID_WTYPE` (e.g. `BW_001_W1`)
| `correct_answer` column is filled in canonical format (see `shasshy.md` format rules)

---

## What disqualifies a variant

- Answer changes from canonical (for W1‚ÄìW4, W6)
- Any number changed, even by accident
- Entity mapping is inconsistent (W3)
- Information dropped (W2)
- LLM used to generate
- Reversal is ambiguous or ill-defined (W5)
- Variant is trivially identical to another variant of the same problem

If you find a disqualifying issue in someone else's variant during review, comment on the PR with the specific problem. Do not silently fix it.

---

## Cross-review rules

- Every variant is reviewed by one person who did not write it
- Reviewer checks the pre-submission checklist item by item
- Reviewer independently solves the problem from the variant text and verifies the answer matches `correct_answer`
- If anything fails: comment on the PR, return to writer, do not merge
- Nandini tracks review assignments and status
